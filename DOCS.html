1 · ภาพรวม

test/ คือสำเนาของแอปที่ ต่อ backend จริง เพื่อทดลองฟีเจอร์ "เพิ่มกล้อง" ให้ใช้งานได้จริง แยกอิสระจาก app/ (ตัวสะอาดสำหรับ GitHub) — แก้ใน test ไม่กระทบ app

เป็นแอปแบบ Desktop (หน้าต่าง native) เปิด UI ที่เป็น HTML/JS ภายใน
ภาษา Python (backend) + HTML/CSS/JS (frontend) คุยกันผ่าน pywebview bridge
เก็บข้อมูล SQLite ไฟล์เดียว tathip.db (กล้อง + เส้นทางที่บันทึก)
ทำงานได้จริง เลือกไฟล์วิดีโอ · ดึงเฟรม · OCR เวลา · วาด ROI · CRUD กล้อง · บันทึกเส้นทางถาวร

4 · สะพาน JS ↔ Python (pywebview)

ตอนสร้างหน้าต่าง เราผูก object Python เข้ากับ JS:

# app.py
webview.create_window("TaThip ...", INDEX, js_api=Api(), width=1340, height=900)
webview.start()

ฝั่ง JS ทุกเมธอดของ Api จะโผล่ใต้ window.pywebview.api และคืนค่าเป็น Promise:

// index.html
const api = () => (window.pywebview && window.pywebview.api) || null;
const info = await api().choose_video();      // เปิด dialog เลือกไฟล์
const r    = await api().read_timestamp(path);  // OCR
Fallback เบราว์เซอร์: ถ้าเปิด index.html ตรงๆ (ไม่มี pywebview) → api() คืน null โค้ดจะสลับไปใช้ mock/in-memory อัตโนมัติ → UI ยังเล่นได้ แค่ไม่เลือกไฟล์จริง/ไม่เก็บถาวร

ตอนแอปพร้อม pywebview จะยิง event pywebviewready เราจึงโหลดข้อมูลจาก DB ตอนนั้น:

if(window.pywebview && window.pywebview.api){ loadState(); }
else{ window.addEventListener("pywebviewready", loadState); }

5 · Database (SQLite)

ไฟล์เดียว tathip.db สร้างอัตโนมัติด้วย init_db() · 2 ตาราง:

CREATE TABLE cameras(
  id TEXT PRIMARY KEY, name, lat REAL, lon REAL,
  video_path, start_time, roi, ts_box, created);   -- roi/ts_box เก็บเป็น JSON string

CREATE TABLE saved_paths(
  id TEXT PRIMARY KEY, name, note, created, sel);  -- sel = JSON ของผลการสืบทั้งชุด
คอลัมน์สำคัญเก็บอะไร
cameras.roiพิกัดมุมโซน ROI (สัดส่วน 0–1) เป็น JSON เช่น [[0.1,0.4],...]
cameras.start_timeเวลาเริ่มคลิป (จาก OCR หรือกรอกเอง) — ใช้ sync เวลาข้ามกล้อง
cameras.video_pathpath วิดีโอ (อ้างอิง ไม่คัดลอกไฟล์)
saved_paths.selJSON: รถเป้าหมาย + track ที่ติ๊ก → เปิดดูซ้ำได้ครบ
ROI/sel เป็นโครงซ้อน → เก็บเป็น JSON string ในคอลัมน์เดียว แล้ว json.loads ตอนอ่าน (เหมาะกับข้อมูลกึ่งโครงสร้าง ไม่ต้องแตกหลายตาราง)

7 · OCR จริง (read_timestamp)

อ่าน timestamp ที่ฝังในคลิป (เช่น 2026/04/05 07:14:55) แบบ 6 ขั้น:

1
ดึงเฟรม
cv2 อ่านเฟรมที่ ~เฟรม 30
2
crop มุมบนซ้าย
50%กว้าง × 13%สูง
3
ขยาย 2×
ตัวเล็ก→อ่านชัดขึ้น
4
EasyOCR
allowlist=ตัวเลข/:./-
5
regex ยืดหยุ่น
anchor ปี 20xx
6
ตรวจช่วงค่า
เดือน1-12, วิ<60…
# regex: ยอมให้ตัวคั่นเป็นอะไรก็ได้/หาย (OCR มักอ่าน ":" หลุด)
TS_RE = r"(20\d{2})\D{0,2}(\d{2})\D{0,2}(\d{2})\D{0,4}(\d{2})\D{0,2}(\d{2})\D{0,2}(\d{2})"

for m in TS_RE.finditer(joined):        # หาคู่แรกที่ "สมเหตุผล"
    y,mo,d,hh,mm,ss = map(int, m.groups())
    if 1<=mo<=12 and 1<=d<=31 and hh<24 and mm<60 and ss<60: ...
กล้องผล OCR จริงสถานะ
cam102026/04/05 07:14:55 (ตรงเป๊ะ)อ่านได้
cam132026/04/05 07:30:xxอ่านได้
cam12วินาทีเพี้ยน → ช่องให้กรอกเองfallback
ทำไมต้องมี fallback: overlay บางกล้องฟอนต์/เรนเดอร์ทำให้ OCR อ่านวินาทีเพี้ยน → ช่องเวลาในหน้า wizard แก้เองได้ เสมอ กัน cross-camera เพี้ยนเงียบ · คืน box มาวาดกรอบเหลืองบนเฟรมให้เห็นด้วย

8 · Front-end data layer

แยกข้อมูล 2 แหล่งให้ชัด:

ตัวแปรมาจากใช้กับ
CAMERASDB (get_state)sidebar · ตารางกล้อง · มินิแมป · KPI · ตั้งค่า/ลบ
TDATA.tracks / CAMSpipeline (tracks_data.js)ค้นหา (Re-ID) · visualize · filter
savedPathsDBหน้าเส้นทางที่บันทึก
// โหลดจาก DB เมื่อแอปพร้อม แล้ว re-render
async function loadState(){
  const s = await api().get_state();
  CAMERAS    = s.cameras;
  savedPaths = s.saved_paths;
  renderSideCams(); renderCamTable(); dash();
}
ทำไมแยก 2 แหล่ง: "กล้องที่ลงทะเบียน" (ผู้ใช้เพิ่ม) กับ "ผลรถที่ตรวจพบ" (pipeline ผลิต) เป็นคนละเรื่อง — แยกแล้วจัดการกล้องได้โดยไม่ต้องรอ pipeline และ pipeline เติม tracks ทีหลังได้อิสระ

10 · ตั้งค่า / ลบกล้อง

กดปุ่ม "ตั้งค่า" (ใน sidebar หรือตาราง) → เปิด modal แก้ ชื่อ/พิกัด/เวลา หรือลบ:

camSettings(id)      // เปิด modal เติมค่าเดิม
saveCamSettings()    // → api.update_camera(id, fields) + อัปเดต UI
deleteCam()          // → confirm → api.delete_camera(id) + เอาออกจาก CAMERAS
ทั้งหมด sync ทั้ง หน่วยความจำ (UI) และ DB พร้อมกัน → ปิดเปิดแอปใหม่ข้อมูลตรง

11 · ค้นหา & Visualize

ส่วนนี้ทำงานบน TDATA.tracks (จาก pipeline):

  • เลือกเป้าหมาย → คำนวณ cosine similarity จาก embedding Re-ID จริง (เมื่อมีข้อมูล)
  • ติ๊กยืนยัน (human-in-the-loop) → Visualize: หมุดกล้อง + เวลา + ลูกศรวิ่งตามลำดับ + โหมด Pathway
  • บันทึกเส้นทาง → save_path ลง DB → เปิดดูซ้ำได้
ตอนนี้ tracks_data.js ว่าง → ค้นหายังไม่มีผล จนกว่าจะรัน pipeline (ดูข้อ 12)

12 · จริง vs รอทำ

ฟีเจอร์สถานะหมายเหตุ
เลือกไฟล์วิดีโอ + อ่าน fps/ขนาดจริงOpenCV (เดสก์ท็อป)
ดึงเฟรมแรกจากคลิปจริงcv2
OCR อ่านเวลาจริงEasyOCR + fallback กรอกเอง
ปักหมุด / วาด ROIจริงLeaflet / SVG polygon
DB เก็บกล้อง/เส้นทางถาวรจริงSQLite
ตั้งค่า / ลบกล้องจริงupdate/delete + sync DB
ค้นหา Re-ID (cosine)โค้ดจริงรอข้อมูลจาก pipeline
Pipeline (detect→track→Re-ID)รอ modelrun_pipeline.py พร้อม · รอ best.pt จาก Colab → เชื่อมปุ่ม "ประมวลผล"
